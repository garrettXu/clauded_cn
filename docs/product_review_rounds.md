# 双 Agent 产品方案三轮审查

## 审查目标

对“复刻 Agent”和“翻译 Agent”进行三轮产品反思，发现方案缺口，给出修正方案，并把修正落实到最终产品文档中。

## 第一轮：职责边界与交付契约

### 发现的问题

1. 原方案把复刻、翻译、样式修复放在同一条流水线里，排查问题时职责混乱。
2. 复刻 Agent 的输出不够标准化，翻译 Agent 不知道应该依赖哪些文件。
3. 缺少明确验收标准，无法判断“复刻完成”或“翻译完成”。
4. 外部链接、子域名、query 参数的处理规则不够细。

### 反思

样式丢失和没有目标语言内容是两类问题。样式丢失属于复刻链路，目标语言缺失属于翻译链路。如果两个问题混在一个 Agent 中，会导致失败重试、日志定位和后续扩展都变复杂。

### 修正方案

1. 拆分为两个独立 Agent。
2. 复刻 Agent 输出 `mirror/original`。
3. 翻译 Agent 只读取 `mirror/original`，默认输出 `mirror/zh-CN`，也可以输出其他语言目录。
4. 增加标准交付物：
   - `manifest.json`
   - `asset_manifest.json`
   - `link_graph.json`
   - `crawl_report.json`
   - `translation_report.json`
   - `layout_report.json`
5. 增加 MVP 和正式版验收标准。

### 已落实到文档

- [复刻 Agent 产品文档](replication_agent_prd.md)
- [翻译 Agent 产品文档](localization_agent_prd.md)

## 第二轮：失败模式与边界场景

### 发现的问题

1. 现代网站常见 SPA、Next.js、Webflow、懒加载，纯 HTTP 抓取会漏资源。
2. CSS 内部 `url(...)`、`srcset`、favicon、OG 图片等容易遗漏。
3. URL 参数可能造成无限抓取。
4. 视频和附件可能导致磁盘不可控。
5. 翻译模型可能返回非 JSON、漏项、术语不一致。
6. 目标语言文本变长后会导致导航、按钮、卡片、移动端布局错乱。

### 反思

复刻 Agent 的关键不是“能下载 HTML”，而是“本地打开时资源完整、链接通顺”。翻译 Agent 的关键不是“能翻译文本”，而是“目标语言站可读、可用、样式不崩”。

### 修正方案

1. 复刻 Agent 增加双抓取策略：HTTP + 浏览器渲染。
2. 复刻 Agent 增加资源类型清单和完整性校验。
3. 增加 URL 规范化和队列限制。
4. 增加资源大小、类型和下载策略配置。
5. 翻译 Agent 增加翻译缓存、术语表和 JSON 输出校验。
6. 翻译 Agent 增加目标语言排版补丁和视觉校验。

### 已落实到文档

- 复刻 Agent：站点发现、页面抓取、资源下载、完整性校验、状态机。
- 翻译 Agent：翻译范围、缓存、术语表、目标语言排版、视觉校验、自动修复策略。

## 第三轮：自动化、运维与长期可维护性

### 发现的问题

1. 没有明确每日同步如何和翻译增量衔接。
2. 没有断点续跑和失败状态定义。
3. 没有成本控制和缓存策略。
4. 没有安全规则说明密钥、日志和副作用操作。
5. 没有补丁回滚机制，自动修复可能越修越乱。
6. 没有可观测性指标，长期运行难以判断健康状态。

### 反思

这个产品不是一次性脚本，而是长期运行的自动化系统。长期系统必须有状态、报告、缓存、重试、回滚、成本控制和安全边界。

### 修正方案

1. 复刻 Agent 每日同步只输出新增和变化内容。
2. 翻译 Agent 只处理复刻 Agent 标记的新页面和变化页面。
3. 页面和资源都用 hash 判断变化。
4. 翻译结果用 hash 缓存，支持人工修订锁定。
5. 自动 CSS 修复只允许补丁文件，不直接破坏原始 CSS。
6. 密钥只读环境变量或密钥管理，不写日志、不入库。
7. 增加状态机和失败状态。
8. 增加 MVP 与正式版验收指标。

### 已落实到文档

- 两个 Agent 都增加了输入输出契约、状态机、增量处理和验收标准。
- 翻译 Agent 增加了补丁可回滚要求。
- 两个 Agent 都明确禁止有副作用的浏览器操作。

## 最终结论

最终产品方案应按以下顺序开发：

1. 先开发复刻 Agent。
2. 用小站点验证资源完整性、链接重写和本地打开效果。
3. 再开发翻译 Agent。
4. 接入翻译缓存、术语表和中文排版补丁。
5. 最后加入视觉模型校验和每日增量自动化。

原因：

- 复刻完整性是翻译质量的前置条件。
- 样式丢失必须在复刻阶段解决。
- 目标语言排版问题必须在翻译阶段解决。
- 两个 Agent 独立后，失败定位、重试、成本控制和未来扩展都更清晰。

## 复刻 Agent 多域名静态部署追加审查

### 新增要求

用户进一步明确：复刻 Agent 必须完整复刻主域名和主域名下子域名；本地预览时主域名和子域名使用不同端口；部署时生成部署声明和 Nginx 配置；复刻后不需要数据库存储，只需要静态复刻；所有域名下的 path 保持不变，只替换 host。

### 检查结果

原复刻 Agent 文档存在 5 个缺口：

1. 只描述了同主域名抓取，没有明确每个 host 独立输出。
2. 只描述了本地路径改写，没有明确“path 保持不变，只替换 host”。
3. 没有要求主域名和子域名使用不同本地端口。
4. 没有部署声明和 Nginx 配置作为复刻 Agent 的交付物。
5. 总览文档仍保留数据库模型表述，容易误解为复刻站运行依赖数据库。

### 修正方案

已补充：

1. 每个原始 host 独立输出到 `mirror/original/hosts/{host}/`。
2. 每个 host 独立分配本地端口，例如 `www.example.com → localhost:8300`、`docs.example.com → localhost:8301`。
3. 本地预览和线上部署都保持 path 不变，只替换 host。
4. 复刻 Agent 必须生成 `DEPLOYMENT.md`。
5. 复刻 Agent 必须生成 `nginx/local-preview.conf` 和 `nginx/mirror.conf`。
6. 明确复刻产物是纯静态包，不依赖数据库。
7. 新增 `host_manifest.json` 和 `query_manifest.json`，处理多 host 和 query 页面。

### 当前仍需开发阶段验证

- 子域名很多时，端口和 Nginx server block 数量是否需要分组管理。
- query 参数影响内容时，纯静态 Nginx 映射是否足够。
- 原站依赖后端 API 的交互是否需要在复刻站降级提示。
- 大视频和附件是否需要按站点配置启用。
- CDN 必要资源与第三方跟踪脚本要区分处理。

### 结论

复刻 Agent 的产品方案已经从“单站镜像”升级为“多 host 静态复刻与部署系统”。下一步开发应先实现：

1. host 发现与 host 队列。
2. host 到本地端口的映射。
3. path 保持的静态文件落点。
4. 跨 host 链接重写。
5. `DEPLOYMENT.md` 生成。
6. Nginx 配置生成。
